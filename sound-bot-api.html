Discord Sound Upload Bot — API Specification

API-style interface definition for an audio submission bot that validates WAV files and commits them into a Git repo.

1) Overview

The Discord Sound Upload Bot allows contributors (musicians/sound designers) to submit audio assets through Discord. The bot validates uploads (file type/size), normalises naming, routes files into the correct project folder (e.g., SFX/Foley/Voice Acting/Composition), then commits and pushes changes to a Git repository.

Note: In the current prototype, these calls can be internal Python functions (no web server required). This document defines a stable interface/contract that could later be exposed as REST endpoints for dashboards or automation.

2) API Surface

Base URL shown for documentation purposes: https://api.example.com/v1

MethodEndpointPurpose
POST /submissions Create a new audio submission (metadata + file reference).
GET /submissions List/search submissions (by user, category, status).
GET /submissions/{submissionId} Retrieve full submission details (validation + git status).
POST /submissions/{submissionId}/validate Validate file rules (extension, max size, optional sample rate/bit depth checks).
POST /submissions/{submissionId}/stage Route file into correct repo folder and stage git changes.
POST /submissions/{submissionId}/commit Commit changes with a standardised message (includes metadata).
POST /submissions/{submissionId}/push Push commit to remote (supports auth + failure handling).
GET /meta/categories Allowed submission categories (SFX/Foley/Voice Acting/Composition).
GET /health Service health check.

4) API Reference

Create Submission

POST/submissions — Registers a new audio submission with metadata and file reference.

ParameterTypeRequiredDescription
guildIdstringYesDiscord server ID scope.
uploaderUserIdstringYesDiscord user ID of uploader.
categorystringYesOne of: SFX, Foley, Voice Acting, Composition.
assetNamestringYesHuman-friendly name used for slugging/file naming.
descriptionstringNoOptional notes for implementers.
file.urlstringYesDiscord CDN URL or uploaded blob URL.
file.filenamestringYesOriginal file name from uploader.
file.sizeBytesintNoUsed for max-size validation (e.g., 25MB).

Returns

FieldTypeMeaning
submissionIdstringUnique ID for tracking this upload pipeline.
statusstringPipeline status: RECEIVED/VALIDATED/STAGED/COMMITTED/PUSHED/FAILED.
createdAtstringCreation time (ISO-8601).

Errors / Failure cases

  • 400 Bad Request — missing category/assetName/file data.
  • 413 Payload Too Large — file exceeds max allowed size.
  • 415 Unsupported Media Type — extension not allowed (only .wav).
  • 429 Too Many Requests — spam/rate limiting (burst uploads).
  • 500 Internal Server Error — storage failure.

Validate Submission

POST/submissions/{submissionId}/validate — Validates file rules and records results.

Typical checks (depending on implementation):

  • Extension is .wav
  • Size <= 25MB
  • Optional: sample rate/bit depth (e.g., 48kHz/24-bit) via FFmpeg/metadata

Errors

  • 415 unsupported file type
  • 422 Unprocessable Entity corrupted/invalid WAV
  • 504 Gateway Timeout metadata tool/FFmpeg timed out

Stage / Commit / Push

POST/submissions/{submissionId}/stage — Moves file into correct repo folder and stages changes.

POST/submissions/{submissionId}/commit — Creates a commit with standard message format.

POST/submissions/{submissionId}/push — Pushes to remote repository.

Common git-related failure cases:

  • 401 Unauthorized — missing/invalid Git credential (token/SSH).
  • 409 Conflict — remote changed; requires pull/rebase or retry.
  • 423 Locked — repo locked / cannot write (permissions).
  • 500 — git command failed (non-zero exit) or repo path misconfigured.

5) Notes (Implementation Mapping)

  • Discord layer: commands/modals collect category + file upload URL.
  • Validation: extension/size checks; optional audio metadata checks.
  • Routing: maps categories to repo folders (e.g., DropOff/Sound/Sfx).
  • Version control: commits create a traceable audit history for assets.
  • Future: expose the interface as REST to support a dashboard or automated QA validation.
